## Business why (from PM/PO)

Every story in Epic 01 (and every epic after it) assumes a working Next.js 15 App Router project with TypeScript strict, Tailwind, Prisma, shadcn/ui, and a testing harness, arranged in the exact layout ADR-0008 prescribes. Without it, each subsequent story would either have to re-do the setup (ugly diffs, wrong in places) or silently freelance scope (erodes BMAD's single-purpose-story invariant).

Ship this, and every future story gets to open a clean, conventional, linting-green, type-checked, test-running repo and focus on its own AC.

This story is **non-user-facing**. Its value is the platform every subsequent story stands on. The "user" here is James (and future Tushar-who-forgot-how-this-was-set-up).

## User story

**As a** developer

**So that** every subsequent Epic 01+ story can focus on feature work, not setup.

## Acceptance criteria

- [ ] **AC-1:** `npm run dev` boots and `/` renders a blank but styled landing page (placeholder `Welcome — v1 bootstrap OK`) without console errors or hydration warnings.
- [ ] **AC-2:** `npm run lint`, `npm run typecheck`, and `npm test` each exit 0 against the empty scaffold. `npm run test:e2e` exits 0 against a single trivial Playwright smoke test hitting `/`.
- [ ] **AC-3:** `npx prisma validate` and `npx prisma generate` succeed against a minimal `schema.prisma` that contains only the `generator` + `datasource` blocks (no models yet — models land in 01.1). Both `DATABASE_URL` (pooled) and `DIRECT_URL` (direct) are wired per ADR-0003. Both commands are **offline** — they parse the schema and generate the client but do not connect to Postgres; no Supabase project is required at this story. However, Prisma's CLI loads env vars from `.env` at project root, so a gitignored `.env` with placeholder values must exist for the commands to resolve `env("DATABASE_URL")` / `env("DIRECT_URL")` without erroring. `.env.example` is the committed template (empty values); `.env` is never committed.
- [ ] **AC-4:** shadcn/ui is initialized (`components/ui/` populated with a `Button` primitive as proof-of-integration — no other components in this story). Design tokens from front-end-spec.md §3 are present in `app/globals.css` + `tailwind.config.ts`: spacing scale (Tailwind default), radii (6/12/999px), typography scale (display/H1/H2/body/caption — mobile+desktop pairs), Inter font with `font-feature-settings: "tnum"`, and semantic color tokens (`--primary`, `--expense`, `--income`, `--warn`, `--danger`, `--muted`) as CSS variables consumed via Tailwind's theme extension.
- [ ] **AC-5:** `sonner`'s `<Toaster />` is mounted in `app/layout.tsx` (root layout) and visually appears when `toast('test')` is called from a client component. A throwaway test route or `<Button onClick>` proves it; delete the proof before commit.

## UX reference

Not a user-facing story. The only UI surface is the placeholder landing at `/`:

```
┌───────────────────────────────────────────┐
│                                           │
│                                           │
│          Welcome — v1 bootstrap OK         │
│                                           │
│                                           │
└───────────────────────────────────────────┘
```

Centered, Inter font, body-size type, `--muted` foreground. No header, no nav. Story 01.1 replaces this landing with the login flow.

Design tokens from front-end-spec.md §3 are what matters here, not this placeholder. The tokens must be **applied correctly** so that 01.1 can just drop in shadcn primitives and get the right look. Token values:

- **Spacing:** Tailwind default scale — no config override needed.
- **Radii:** `--radius-sm: 6px` (inputs, buttons), `--radius-md: 12px` (cards), `--radius-full: 9999px` (pills). Expose via `tailwind.config.ts` `theme.extend.borderRadius`.
- **Typography scale** (mobile / desktop, rem-based):
  - `display`: 2rem / 2.5rem
  - `h1`: 1.5rem / 1.75rem
  - `h2`: 1.125rem / 1.25rem
  - `body`: 0.9375rem / 1rem
  - `caption`: 0.8125rem / 0.875rem
- **Font:** Inter via `next/font/google`, with `font-feature-settings: "tnum"` applied globally on `<body>` for tabular numerals.
- **Color tokens** — semantic names map to HSL CSS variables; provide **light-mode values now** and stub dark-mode with the same tokens reassigned in `[data-theme='dark']` (values can be placeholder — dark-mode polish is v1.5 per spec §3):

  ```css
  :root {
    --background: 0 0% 100%;
    --foreground: 222 47% 11%;
    --primary: 239 84% 60%;          /* brand indigo */
    --primary-foreground: 0 0% 100%;
    --expense: 0 60% 48%;             /* desaturated red */
    --income: 142 55% 38%;            /* desaturated green */
    --warn: 38 92% 50%;               /* amber */
    --danger: 0 72% 51%;              /* destructive red */
    --muted: 215 16% 47%;
    --muted-foreground: 215 20% 65%;
    --border: 214 32% 91%;
    --input: 214 32% 91%;
    --ring: 239 84% 60%;
  }
  ```

  Expose these to Tailwind via `theme.extend.colors` using `hsl(var(--primary))` syntax so classes like `bg-primary`, `text-expense`, `border-border` work.

Full context if you want it: `bmad/artifacts/front-end-spec.md §3 Design tokens`.
