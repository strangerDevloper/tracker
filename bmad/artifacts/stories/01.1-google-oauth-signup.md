# Story 01.1: Google OAuth sign-in

James — read this file start to finish. You should not need to open the PRD, the architecture doc, the front-end spec, or any ADR to complete this story. If you find yourself wanting to, pause and tell me — something is missing from this file and I'll fix it.

## Business why (from PM/PO)

Every feature of the app — transactions, investments, budgets, dashboards — is scoped by `user_id`. Until auth works, nothing else can be built safely, because Row-Level Security depends on a known authenticated user. Google OAuth is the happy path for 80% of users — one click, no passwords — so it's the first login method we wire up. Magic link (01.2) and email/password (01.3) share this story's user-upsert and callback plumbing, so getting this one right makes those two cheap.

Ship this, and every subsequent epic gets to assume `getAuthedUser()` works and that RLS will enforce isolation for free.

## User story

**As a** user
**I want** to sign in with my Google account
**So that** I don't have to create and remember another password.

## Acceptance criteria

- [ ] **AC-1:** Clicking "Sign in with Google" on `/login` initiates the Supabase OAuth flow and redirects to Google's consent screen.
- [ ] **AC-2:** After consent, I land on `/dashboard` (authenticated) — session cookies set, subsequent requests to `(finance)` routes succeed.

- [ ] **AC-6:** Integration test proves `public.users.id === auth.uid()` after signup — run a server-side query with the new session's JWT and confirm `SELECT id FROM public.users WHERE id = auth.uid()` returns exactly one row for the signed-in user.

## Data model impact

Two new tables. Do this as a single Prisma migration called `init_users_settings`.

```prisma
// prisma/schema.prisma
model User {
  // CRITICAL: no @default. The id is set explicitly on upsert to auth.users.id
  // (the Supabase-issued uuid). All RLS policies rely on `user_id = auth.uid()`.
  // Letting Prisma generate a fresh uuid here would silently break data isolation
  // for every subsequent epic. See alignment-2026-04-24.md Conflict #3.
  id            String   @id
  email         String   @unique
  name          String?
  authProvider  String   @map("auth_provider")
  createdAt     DateTime @default(now()) @map("created_at")

  settings      UserSettings?

  @@map("users")
}

model UserSettings {
  userId              String   @id @map("user_id")
  currency            String   @default("INR")
  locale              String   @default("en-IN")
  timezone            String   @default("Asia/Kolkata")
  fyStartMonth        Int      @default(4)  @map("fy_start_month")
  notificationPrefs   Json     @default("{}") @map("notification_prefs")

  user                User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@map("user_settings")
}
```

### Migration SQL — additions beyond Prisma's auto-generated DDL

Prisma will generate the `CREATE TABLE` statements. You need to add two things to the generated `migration.sql`:

1. **Foreign key from `public.users.id` to `auth.users.id`** — so a user can't exist in `public.users` without a corresponding `auth.users` row. Prisma doesn't know about `auth.users` (it's not in the Prisma schema), so this FK is added as raw SQL.
   ```sql
   ALTER TABLE public.users
     ADD CONSTRAINT users_id_fkey
     FOREIGN KEY (id) REFERENCES auth.users(id) ON DELETE CASCADE;
   ```
2. **Enable RLS on both tables** (policies themselves are Story 01.5, but enabling RLS is harmless and cheap to do now):
   ```sql
   ALTER TABLE public.users ENABLE ROW LEVEL SECURITY;
   ALTER TABLE public.user_settings ENABLE ROW LEVEL SECURITY;
   ```
   Don't add policies in this migration — Story 01.5 owns the full policy set. Without policies, RLS-enabled tables deny all access from the anon/authenticated role, which is fine because this story only touches them via the server role during callback.

### `auth.users` access

Supabase manages `auth.users`. You cannot add columns to it and should not query it directly from app code — always go through `supabase.auth.getUser()` from the SSR helper.

## API contract

### `GET /api/auth/callback`

Handles both Google OAuth callback and (future) magic-link callback.

**Query params** (set by Supabase's redirect):
- `code: string` — OAuth authorization code.
- (Optionally) `next: string` — path to redirect to post-auth. Default `/dashboard`.

**Behavior:**
1. Exchange `code` for a session via `supabase.auth.exchangeCodeForSession(code)`.
2. Read the authenticated user via `supabase.auth.getUser()`.
3. Call `upsertUserOnLogin(user)` — the helper that writes `public.users` and `public.user_settings`.
4. Redirect to `next` (default `/dashboard`) via `NextResponse.redirect()`.

**Error behavior:**
- Missing/invalid `code` → redirect to `/login?error=auth_failed`.
- Exchange failure → log to Sentry, redirect to `/login?error=auth_failed`.
- User-upsert failure → log to Sentry, redirect to `/login?error=account_setup_failed`. (User's Supabase session still exists — they can retry.)

### `GET /api/me`

Returns the current user + settings. Used by the app shell (Story 01.6) but scaffold it now — trivial.

**Response 200:**
```json
{
  "data": {
    "user": { "id": "...", "email": "...", "name": "...", "authProvider": "google" },
    "settings": { "currency": "INR", "locale": "en-IN", "timezone": "Asia/Kolkata", "fyStartMonth": 4 }
  }
}
```

**Response 401:** `{ "error": { "code": "UNAUTHENTICATED", "message": "Sign in required" } }`

### Error response shape (project-wide convention)

```ts
// Success
{ "data": T }
// Error
{ "error": { "code": "VALIDATION_ERROR" | "UNAUTHENTICATED" | "NOT_FOUND" | "CONFLICT" | "INTERNAL", "message": string, "details"?: any } }
```

## Tech guidance (from Architect)

### Supabase Auth with Next.js App Router

Use `@supabase/ssr` — **not** `@supabase/auth-helpers-nextjs` (deprecated). Follow the canonical pattern:

```ts
// lib/auth/server.ts  — use in Server Components, Route Handlers, Server Actions
import { createServerClient } from '@supabase/ssr';
import { cookies } from 'next/headers';

export function getSupabaseServerClient() {
  const cookieStore = cookies();
  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        get: (name) => cookieStore.get(name)?.value,
        set: (name, value, options) => cookieStore.set({ name, value, ...options }),
        remove: (name, options) => cookieStore.set({ name, value: '', ...options }),
      },
    }
  );
}
```

### The upsert helper — the single most important code in this story

```ts
// lib/auth/upsert-user-on-login.ts
import type { User as AuthUser } from '@supabase/supabase-js';
import { prisma } from '@/lib/db/client';

export async function upsertUserOnLogin(authUser: AuthUser): Promise<void> {
  // MUST pass id explicitly. Do not omit it — Prisma would then try to generate
  // a new uuid (if a @default were set) or error (without one). Either way, it
  // would be wrong — we need public.users.id = auth.users.id for RLS.
  await prisma.user.upsert({
    where: { id: authUser.id },
    update: {
      email: authUser.email!,
      name: authUser.user_metadata?.full_name ?? authUser.user_metadata?.name ?? null,
    },
    create: {
      id: authUser.id, // <-- mirroring auth.users.id; non-negotiable
      email: authUser.email!,
      name: authUser.user_metadata?.full_name ?? authUser.user_metadata?.name ?? null,
      authProvider: detectProvider(authUser),
      settings: {
        create: {}, // all defaults: INR / en-IN / Asia/Kolkata / fy_start_month=4
      },
    },
  });
}

function detectProvider(u: AuthUser): string {
  const identities = u.identities ?? [];
  const google = identities.find((i) => i.provider === 'google');
  if (google) return 'google';
  const email = identities.find((i) => i.provider === 'email');
  if (email) return 'email'; // magic link and password both report 'email'
  return 'unknown';
}
```

### `getAuthedUser()` — the helper every future feature will use

Scaffold this now so 01.2+ can reuse it:

```ts
// lib/auth/get-authed-user.ts
import { getSupabaseServerClient } from '@/lib/auth/server';
import { AppError } from '@/lib/errors/app-error';

export async function getAuthedUser() {
  const supabase = getSupabaseServerClient();
  const { data: { user }, error } = await supabase.auth.getUser();
  if (error || !user) throw new AppError('UNAUTHENTICATED', 'Sign in required', 401);
  return user; // the Supabase auth.users row
}
```

`AppError` is a tiny class carrying `code`, `message`, `status`. If `lib/errors/app-error.ts` doesn't exist yet, create it with a minimal implementation:

```ts
// lib/errors/app-error.ts
export class AppError extends Error {
  constructor(public code: string, message: string, public status = 500) {
    super(message);
  }
}
```
